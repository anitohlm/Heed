## Bugs Found

### B-01 — AddPlanSheet: Stale State on Reopen

**Severity:** Medium  
**Component:** `AddPlanSheet`  
**Repro:**
1. Open "+ Add plan" → choose Goal → fill in a title and target date.
2. Tap backdrop (outside sheet) to dismiss without submitting.
3. Open "+ Add plan" again.

**Observed:** Previous title, target date, goalKind, tasksText, unit, etc. still populated.  
**Expected:** Sheet opens fresh each time the user hasn't submitted.  
**Root cause:** No `useEffect(() => { /* reset */ }, [open])` hook in `AddPlanSheet`. State is only cleared inside `handleSubmit` (line 3919). The `onClose` path does not reset state.  
**Impact:** User may accidentally create a plan with leftover data from a previous abandoned session.

---

### B-02 — `type === null` Crash Risk in `handleSubmit`

**Severity:** Medium  
**Component:** `AddPlanSheet.handleSubmit` (line 3913)  
**Code:**
```js
const plan = { ..., icon: PLAN_TYPES.find(p => p.type === type).icon, ... }
```
**Repro:** Theoretically unreachable in normal flow (the form step only renders when `type && step === 'form'`), but if `handleSubmit` is ever triggered with `type === null` (e.g., rapid state change, future refactor), `find()` returns `undefined` and `.icon` throws a TypeError, crashing the component.  
**Expected:** Guard with `if (!type) return` before the find call.  
**Impact:** Low probability today; high blast radius if hit (uncaught render crash).

---

### B-03 — `upcomingRemaining` Includes Tasks Due Today (dueIn === 0)

**Severity:** Low  
**Component:** `TodayTab` (line 2799)  
**Code:**
```js
const upcomingRemaining = remainingTasks.filter(t => t.dueIn !== undefined)
```
**Issue:** `dueIn === 0` means "due today" and passes this filter. A task due today with a non-critical score could appear in the "Upcoming" section below the fold while also being eligible for the top 3 focus tasks — creating a potential duplicate display if the focus slots are occupied by higher-scored tasks.  
**Expected:** Filter should be `t.dueIn !== undefined && t.dueIn > 0` (or a separate "due today" bucket).  
**Impact:** Edge case — only triggers when there are 4+ tasks and a `dueIn === 0` task ranks outside top 3.

---

### B-04 — Negative Target Amount Allowed for Numeric Goals

**Severity:** Low  
**Component:** `AddPlanSheet` (line 3984)  
**Code:**
```jsx
<input type="number" ... placeholder="50000" value={targetAmt} onChange={e => setTargetAmt(e.target.value)}/>
```
**Issue:** No `min="0"` attribute. The `handleSubmit` guard only blocks `<= 0` (line 3911):
```js
if (type === 'goal' && goalKind === 'numeric' && (parseFloat(targetAmt) || 0) <= 0) return
```
This means `-5000` is blocked (since -5000 <= 0), but `0.001` is allowed.  
**Actual gap:** Target of `0.001` passes validation, creating a goal that reads "100% saved" immediately.  
**Expected:** `min="1"` attribute + validation `target < 1` blocks this.  
**Impact:** Cosmetic; user can create meaningless goals.

---

### B-05 — `showImportanceInfo` Tooltip Not Reset When AddTaskModal Closes

**Severity:** Low  
**Component:** `AddTaskModal` (line 5096)  
**Code:**
```js
useEffect(() => { if (!open) setConfirmingDelete(false) }, [open])
```
**Issue:** The close effect only resets `confirmingDelete`. If the user opens the importance tooltip (`?` button), then closes the modal without dismissing the tooltip, the tooltip state (`showImportanceInfo = true`) persists. The next time the modal opens, the tooltip is already visible.  
**Expected:** Add `setShowImportanceInfo(false)` to the close effect.  
**Impact:** Minor UX: tooltip briefly flashes open on re-entry, which looks like a bug to the user.

---

### B-06 — Existing Goals Without `goalKind` Field Render Inconsistently

**Severity:** Low  
**Component:** `PlanCard`, `GoalUpdateSheet` (various)  
**Issue:** Users who had goal plans saved to localStorage before the `goalKind` field was introduced (in the product-gap-fix feature) have plans without `goalKind`. The code checks `plan.goalKind === 'milestone'`, which is `false` when `goalKind` is `undefined`, so old numeric goals accidentally take the numeric rendering path — which is correct behavior.  
**However:** There is no explicit migration or default. If the logic ever changes to `plan.goalKind === 'numeric'` (a positive check), all legacy plans would silently break.  
**Expected:** Either a migration that defaults missing `goalKind` to `'numeric'` at read time, or a fallback `const goalKind = plan.goalKind ?? 'numeric'` pattern throughout the components.  
**Impact:** Safe today; fragile under future refactors.

---

### B-07 — `suggestDismissed` Not Reset When AddPlanSheet Closes

**Severity:** Low  
**Component:** `AddPlanSheet` (line 3793)  
**Issue:** Related to B-01. If the user dismisses the heuristic task suggestions (taps ✕ on the suggestion chips), then closes the sheet without submitting, the `suggestDismissed = true` state persists on reopen. Suggestions will not reappear even though it's a fresh open.  
**Impact:** Part of B-01's broader stale-state problem.

---

### B-08 — `GoalUpdateSheet` Input Has No `min` Constraint

**Severity:** Low  
**Component:** `GoalUpdateSheet` (line 4076)  
**Code:**
```jsx
<input type="number" value={val} ... />
```
**Issue:** The `submitNumeric` function correctly blocks `n < 0`, but the input field itself has no `min="0"`, so the native number spinner allows navigating to negative values before the user taps Save. On mobile this can be confusing.  
**Expected:** Add `min="0"` attribute to match the runtime guard.  
**Impact:** Cosmetic / confusing UX.
